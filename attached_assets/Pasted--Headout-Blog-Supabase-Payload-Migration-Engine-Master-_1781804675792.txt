# Headout Blog → Supabase → Payload Migration Engine (Master Prompt)

## Mission

You are a senior staff-level software engineer, solutions architect, data engineer, and web crawling expert.

Your goal is to build a **production-grade content migration pipeline** that crawls the entire Headout Blog ecosystem, preserves every piece of content and metadata with zero loss, stores it in a normalized Supabase database, and prepares it for seamless migration into Payload CMS.

This is **NOT** a scraping project.

This is a **content preservation and migration platform**.

Every architectural decision should optimize for **long-term maintainability, data integrity, idempotency, and Payload compatibility**.

---

# Primary Goal

Create a canonical content repository that becomes the single source of truth for future migration into Payload CMS.

Supabase is the source of truth.

The crawler is only an ingestion mechanism.

The final database should contain enough structured information to completely recreate every page without needing to crawl again.

---

# Think Like a CMS, Not a Scraper

Do not store only HTML.

Understand page semantics.

Convert pages into reusable content blocks.

Normalize relationships.

Extract reusable components.

Preserve hierarchy.

Preserve ordering.

Preserve metadata.

Preserve relationships.

Preserve SEO.

Preserve accessibility.

Preserve internal linking.

Preserve rich text formatting.

Preserve content fidelity.

Never optimize for speed at the expense of completeness.

# Crawl Engine

Use Playwright.

Never rely solely on raw HTTP requests.

For every page:

* wait for network idle
* execute JavaScript
* scroll entire page
* trigger lazy loading
* wait for deferred content
* expand accordions
* reveal hidden sections
* wait for hydration
* ensure DOM stability

Extract only after rendering is complete.

---

# Crawl Queue

Maintain a persistent crawl queue.

Each URL should track:

* status
* priority
* retry_count
* first_seen
* last_attempt
* completed_at
* checksum
* content_hash
* crawl_version

Support pause and resume.

Support incremental recrawls.

Support crash recovery.

Never restart from zero.

---

# Idempotency

The crawler must be idempotent.

Running it multiple times must never create duplicates.

Use:

* canonical URL
* content hash
* checksum
* upserts

Detect changed pages.

Update existing records.

Create new versions when content changes.
