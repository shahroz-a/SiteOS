# PRD: Headout Blog Studio – Integrated Internal CMS & Publishing Platform

# Vision

Build a world-class **enterprise content management platform** for the migrated Headout Blog that is **fully integrated into the existing blog application**.

This is **not** a standalone admin panel.

This is **not** a separate CRUD application.

This is **not** a separate website.

It must be a seamless extension of the existing Replit blog project, sharing the same codebase, database, design system, rendering engine, and components.

The platform should combine the best ideas from:

* Payload CMS
* Notion
* Sanity
* Contentful
* Airtable
* Webflow CMS

while preserving Headout's design language through the Eevee Design System.

The platform should enable editors, writers, SEO teams, designers, translators and administrators to create, edit, review, preview and publish content while preserving **100% URL integrity, SEO authority and content fidelity.**

---

# Core Architecture

## Single Application

There should be only **one application**.

```
Headout Blog
│
├── Public Blog
│
└── Internal CMS
```

Both should share:

* one codebase
* one PostgreSQL database
* one API layer
* one authentication system
* one design system
* one rendering engine
* one component library
* one content model

The CMS must never become a separate project.

---

# Single Source of Truth

The PostgreSQL database is the canonical source of truth.

The public blog reads from it.

The CMS reads from it.

The CMS writes to it.

There should never be duplicated content.

Editing from the CMS should immediately affect the same content consumed by the public blog.

# URL Management (Critical)

The existing URL structure is a business and SEO asset.

Example:

```
https://www.headout.com/blog/thanksgiving-vacation-ideas-for-families/
```

must always remain exactly:

```
https://www.headout.com/blog/thanksgiving-vacation-ideas-for-families/
```

Never regenerate slugs automatically.

Never silently change URLs.

Display:

* Original URL
* Current URL
* Canonical URL
* Redirect Status
* SEO Impact

Changing URLs requires elevated permissions and explicit confirmation.

A non-technical editor should be able to:

* log in
* create a blog
* edit a blog
* duplicate a blog
* manage authors
* manage categories
* manage tags
* preview changes
* validate SEO
* preserve original URLs
* schedule publishing
* review history
* restore versions

The public blog and the internal CMS should operate as **one unified platform**, sharing the same data, components, rendering engine and design system, while preserving **100% of content fidelity, URLs, SEO metadata, internal linking, accessibility, component hierarchy and long-term scalability**.
