### Scheduled redirect-health (Scheduled Deployment)

The redirect target-health cleanup only delivers full value when run *repeatedly* over time: off-blog targets are only deactivated once consecutive confirmed-dead readings accumulate to `OFF_BLOG_DEAD_THRESHOLD` (2) across separate runs. Run it as a **Replit Scheduled Deployment** so dead destinations actually get retired without anyone remembering to run it. The agent cannot create the deployment — set it up once from the **Publish** pane (Scheduled type), then it runs itself.

- **Deployment type:** Scheduled. **Schedule:** daily is a sensible cadence (e.g. `0 6 * * *`); each run is one increment toward the off-blog corroboration threshold, so ~2 days to retire a confirmed-dead off-blog target.
- **Build command:** `pnpm install && pnpm --filter @workspace/scripts run build:jobs`
- **Run command:** `pnpm --filter @workspace/scripts run redirect:health:prod` (runs from the `scripts/` package dir so the `reports/` report path resolves to the repo root). Defaults are already gentle on the target origin (concurrency 5, per-probe timeout 15s); tune with `-- --concurrency=N --timeout=MS` if needed. Use `-- --dry-run` for a no-write preview run.
- **Env:** `DATABASE_URL` must point at the **production** Postgres so deactivations persist where the live blog reads them.
- **Observability:** every deactivation is written to `crawl_logs` (a `warn` line, durable in the prod DB) and the full run summary — deactivated + off-blog "at-risk" rows — is printed to stdout, visible in the deployment logs (`fetchDeploymentLogs`). The `redirect-deactivations.json` report is also written, but a scheduled container's filesystem is ephemeral, so treat `crawl_logs` + the stdout summary as the retained record. Each deactivation is reversible by flipping `redirects.isActive` back to true.

### Scheduled page-views rollup (Scheduled Deployment)

The raw `page_views` event log grows one row per article view, unbounded by traffic — left alone it slows the analytics aggregates and bloats the DB. The rollup job (`scripts/src/rollup-page-views.ts`) keeps it bounded by folding every completed past day into the `page_view_daily` rollup and deleting those raw rows. Run it as a **Replit Scheduled Deployment** so storage stays bounded without anyone remembering to run it. The agent cannot create the deployment — set it up once from the **Publish** pane (Scheduled type).

- **Deployment type:** Scheduled. **Schedule:** daily (e.g. `0 4 * * *`); each run rolls up all completed days, so a daily cadence keeps raw rows to ~one day plus whatever current-day traffic has accumulated.
- **Build command:** `pnpm install && pnpm --filter @workspace/scripts run build:jobs`
- **Run command:** `pnpm --filter @workspace/scripts run views:rollup:prod`. Use `-- --dry-run` for a no-write preview, or `-- --retention-days=N` to keep the most recent N completed days as raw event rows.
- **Env:** `DATABASE_URL` must point at the **production** Postgres so the rollup persists where the analytics endpoint reads it.
- **Observability:** every run that actually folds rows writes a null-actor `audit_logs` row (action `analytics.rollup.auto`, entityType `analytics`, the rows/days/buckets/cutoff summary in `after`) so editors see the automated maintenance run in the CMS audit log (`artifacts/cms/src/pages/audit-log.tsx` renders it via `RollupRun` and registers the action/entity in `ACTION_OPTIONS`/`ACTION_LABELS`/`ENTITY_OPTIONS`), plus a durable `crawl_logs` info line (survives the ephemeral scheduled container). No-op runs (nothing eligible) stay quiet. Both DB writes are best-effort and never fail the job. The run summary (rows folded, days, buckets, cutoff) also prints to stdout, visible in the deployment logs (`fetchDeploymentLogs`). The rollup is non-destructive of view *counts* — totals are preserved exactly in `page_view_daily`.

### Scheduled auto-publish of scheduled posts (Scheduled Deployment)

An editor can move a post into `scheduled` with a future `scheduledFor`, but nothing flips it to `published` once that time arrives unless the always-on api-server is up (it runs an in-process 60s scheduler — `publishDueScheduledPosts` in `artifacts/api-server/src/lib/cms-publishing.ts`, wired in `index.ts`). The standalone job (`scripts/src/publish-scheduled.ts`) provides the same guarantee as a **Replit Scheduled Deployment** so scheduling is hands-off even when the server is asleep — finding every `scheduled` post with `scheduledFor <= now`, transitioning it to `published`, stamping `publishedAt` with the originally-scheduled time, and clearing `scheduledFor` in one atomic, idempotent UPDATE. This is what makes the content dashboard's "overdue by X" badge stop appearing. The agent cannot create the deployment — set it up once from the **Publish** pane (Scheduled type). The script logic is intentionally a small re-implementation (scripts are leaf packages and can't import from `artifacts/api-server`); the shared truth is the `pages` schema + the publish state machine.

- **Deployment type:** Scheduled. **Schedule:** frequent is fine (e.g. `*/5 * * * *` every 5 min, or hourly) — the closer the cadence, the smaller the "overdue" window before a due post goes live. Each run is idempotent and a no-op when nothing is due.
- **Build command:** `pnpm install && pnpm --filter @workspace/scripts run build:jobs`
- **Run command:** `pnpm --filter @workspace/scripts run publish:scheduled:prod`. Use `-- --dry-run` for a no-write preview (lists the due posts without publishing).
- **Env:** `DATABASE_URL` must point at the **production** Postgres so publishes land where the live blog and CMS read them.
- **Observability:** each auto-publish writes an `audit_logs` row (action `article.publish.scheduled`, no human actor, before/after status) for the CMS audit trail AND a durable `crawl_logs` info line (survives the ephemeral scheduled container); the per-post outcome + count also print to stdout (`fetchDeploymentLogs`). Both DB writes are best-effort and never fail the job.

### Scheduled full content backup verification (Scheduled Deployment)

The exhaustive full-corpus export→import→backup→restore round-trip (`artifacts/api-server/src/lib/__tests__/cms-io.integration.test.ts`, gated on `VERIFY_CMS_IO=1`) proves no article loses an image, link, or metadata field across the CMS export/import/backup/restore path. The everyday `verify-cms-io` validation gate only checks a 3-page sample (`CMS_IO_VERIFY_LIMIT=3`) because the full ~3.7k-page corpus is too slow for a gate; the full run only happened when someone remembered to trigger `verify:cms-io:full` by hand. Now that the full run is **batched and memory-safe** (`checkBatch` per batch of `CMS_IO_BATCH_SIZE` pages, default 200 — peak memory tracks one batch, not the whole corpus), it can run unattended over every article. Run it as a **Replit Scheduled Deployment** (e.g. weekly) so import/restore regressions are caught corpus-wide before they bite a real export/restore. The agent cannot create the deployment — set it up once from the **Publish** pane (Scheduled type). Unlike the redirect-health/rollup/publish jobs this is a **vitest run, not an esbuild node bundle** (a test runner can't be bundled), so the build step is just `pnpm install` (which provides the root `vitest` dev dependency) — there is no `:prod` bundle variant and no `build:jobs` step.

- **Deployment type:** Scheduled. **Schedule:** weekly is a sensible cadence (e.g. `0 5 * * 0`) — the full corpus takes a while, and import/restore regressions are introduced by code changes, not traffic, so a periodic full sweep is enough; the per-PR `verify-cms-io` 3-page gate covers the fast path.
- **Build command:** `pnpm install` (installs vitest + workspace packages; vitest resolves the `@workspace/*` deps from TS source, so no lib build/`build:jobs` is needed).
- **Run command:** `pnpm --filter @workspace/api-server run verify:cms-io:full` (sets `VERIFY_CMS_IO=1` and runs the whole corpus, no `CMS_IO_VERIFY_LIMIT`). Tune memory vs. per-batch overhead with `CMS_IO_BATCH_SIZE=N` (default 200; lower it — e.g. `CMS_IO_BATCH_SIZE=100` — if the scheduled container OOMs on many-image pages).
- **Env:** `DATABASE_URL` should point at the **production** Postgres (or a clone). The check is **non-destructive by construction** — the export leg only SELECTs and every import runs inside an always-rolled-back transaction, so it never mutates the DB even against prod.
- **Observability:** the run is pass/fail — vitest exits non-zero and prints the failing page's slug + the lost image/link/metadata assertion to stdout, visible in the deployment logs (`fetchDeploymentLogs`); a non-zero scheduled run = a real import/restore regression somewhere in the corpus. There are no DB-side audit rows (the job writes nothing), so the deployment log is the record.

### Scheduled SEO report generation (Scheduled Deployment)

The content explorer's Validation drill-down surfaces the LATEST `validation_reports` row per page, but `seo`-type reports are only written when an editor trips the publish gate (`runPublishGate`) — so as editors publish and edit articles the stored `seo` reports go stale and the drill-down drifts from reality. The batch job (`artifacts/api-server/src/lib/generate-seo-reports.ts`, script `seo:reports`) closes that gap: it walks every published post, runs the SAME validation the publish gate / SEO panel run (`runValidation` → `serializeCmsPostDetail` + `buildValidationInput` + DB duplicate detection + `validateSeo`) and persists a fresh `seo` report via `storeReport`. Left as a one-shot manual run it only refreshes when someone remembers, so run it as a **Replit Scheduled Deployment** to keep corpus SEO health current hands-off. The agent cannot create the deployment — set it up once from the **Publish** pane (Scheduled type). Unlike the redirect-health/rollup/publish jobs (which live in `scripts/`) this job lives in `artifacts/api-server` because it depends on the api-server-only validation glue (`runValidation`/`storeReport` in `seo-validation.ts`), so it is bundled by the api-server's own `build.mjs` (a second entry point alongside `index.ts`, emitted to `dist/lib/generate-seo-reports.mjs`) rather than `scripts/`'s `build:jobs`.

- **Deployment type:** Scheduled. **Schedule:** daily is a sensible cadence (e.g. `0 3 * * *`); SEO reports drift with edits/publishes (not traffic), so a daily refresh keeps the drill-down close to reality. Each run is idempotent (`validation_reports` is append-only; the explorer reads the latest row per page) — it just refreshes the latest `seo` report for every page.
- **Build command:** `pnpm install && pnpm --filter @workspace/api-server run build` (the api-server build bundles both the server and the SEO-report job to `dist/`).
- **Run command:** `pnpm --filter @workspace/api-server run seo:reports:prod` (runs the prebuilt `dist/lib/generate-seo-reports.mjs` with plain `node` — no `tsx`/pnpm needed at runtime, same convention as the scripts/* `:prod` jobs). Use `-- --dry-run` for a no-write preview, `-- --status=draft` to scope to a different page status (default `published`), or `-- --limit=N`/`-- --offset=N` to run the corpus in chunks.
- **Env:** `DATABASE_URL` must point at the **production** Postgres so the refreshed reports land where the CMS content explorer reads them.
- **Observability:** the run summary (pages validated, pass/warn/fail tally, reports written, missing) prints to stdout, visible in the deployment logs (`fetchDeploymentLogs`). The job persists fresh `seo` rows into `validation_reports` (the durable record the explorer reads); a dry-run writes nothing. There are no audit-log rows for this job.

- Redirect target-health (source of truth): `scripts/src/prerender/redirect-target-health.ts` holds TWO complementary, side-effect-free cores that share path/scope helpers. (1) **Auto-deactivation policy** — `targetKind` / `readingVerdict` / `decideHealth` / `normalizeTargetPath` (`OFF_BLOG_DEAD_THRESHOLD=2`, dead = HTTP 404/410); runner CLI `scripts/src/redirect-health.ts` (`pnpm --filter @workspace/scripts run redirect:health`, flags `--dry-run` / `--no-network` / `--concurrency=` / `--timeout=`). It checks every ACTIVE redirect's TARGET (not its fromPath) and flips `redirects.isActive=false` for confirmed-dead ones. **On-blog** (`/blog/...`) targets are deterministic — checked for existence against the served page corpus (all pages' `pathname` in ANY status, so a held-back draft is "pending" not dead, plus `/blog/category/<slug>`, `/blog/author/<slug>`, `/blog/`) and deactivated on the first miss. **Off-blog** targets are an HTTP HEAD/GET probe (follows redirects, via `redirectTargetUrl`) and need corroboration: consecutive confirmed-dead readings must accumulate across runs to `OFF_BLOG_DEAD_THRESHOLD` (`redirects.target_check_failures`); any healthy reading resets the counter, a timeout/network error (`status=null` → `unknown`) is ignored. Auditable/reversible: each row records `deactivated_reason`/`deactivated_at`/`target_last_status`/`target_checked_at`, a `crawl_logs` warn line is written, and a `redirect-deactivations.json` report (deactivated + off-blog "at-risk" below threshold) is emitted. (2) **Verification digest** (DB-free + network-free) — `checkRedirectTargets(redirects, deps)` with injected `onBlogExists`/`probe`, plus `formatHealthDigest` (→ `{subject,text}`, the shared body for BOTH webhook and email so they can't drift), `targetScope`, `totalBroken`, `shouldNotify`, and `buildOnBlogPathSet`/`onBlogExistsIn`. On-blog (`/blog/...`) targets are verified against published content (the static SPA returns 200 for any path, so HTTP-probing them is mean...
- Off-blog redirect preservation: redirects whose OLD url lives outside `/blog/` (e.g. `/statue-of-liberty-cruises-c-121/` → `/statue-of-liberty-tickets-c-121/`) are genuine renames the static blog can never serve (it owns only `/blog/`). **Decision: the blog deployment does NOT own non-`/blog/` redirects — they belong solely to the main Headout site.** They are still preserved, not dropped: `recordRedirects` (`scripts/src/crawler/store.ts`) persists BOTH blog-serveable (`/blog/…`) and off-blog rows to the `redirects` table (discriminated purely by the `/blog/` prefix; no schema column), and is called from `storePage` AND from the pipeline's off-blog skip branch (`scripts/src/crawler/pipeline.ts`) so a blog URL that 301s straight off the blog still records its forwarding redirect. **The off-blog skip branch is the real-world source of off-blog renames** — `assemblePage` only keeps clean *on-blog* hops in a stored page's `redirectChain` (off-blog and malformed hops are captured into `droppedRedirects` there via `classifyRedirectHop`, since an off-blog `from` inside a stored blog page's own chain is anomalous), so off-blog renames reach `recordRedirects` through the skip branch's RAW chain, not through the assemble path. `recordRedirects` itself is prefix-agnostic and persists any off-blog hop it is given. `normalizeRedirectPath` (prefix-agnostic clean-path validator) and `isBlogRedirectPath` live in `redirects.ts`; `normalizeRedirectFromPath` now layers the blog-serveable gate on top of `normalizeRedirectPath`. The prerender naturally skips off-blog rows (`buildRedirectStub` returns null for them). `reports.ts` splits the `redirect-map.json` counts (`blogServeable`/`offBlog`) and emits a dedicated, actionable `off-blog-redirects.json` for the main site's redirect config. (Distinct from `redirect-dropped.json` above, which lists junk hops that never reached the `redirects` table at all.)

- CMS round-trip regression test (opt-in): `artifacts/api-server/src/lib/__tests__/cms-io.integration.test.ts` — export→import→backup→restore against the live **dev** DB, every import inside an always-rolled-back transaction (non-destructive), asserting posts match (none created), re-import is idempotent, and no image/link/metadata is lost. Gated on `VERIFY_CMS_IO=1`. **Registered as the `verify-cms-io` validation step** (in `.replit`) so import/restore regressions are caught automatically — a failing validation run = an import/restore regression. The validation runs `pnpm --filter @workspace/api-server run verify:cms-io`, which is **bounded** (`CMS_IO_VERIFY_LIMIT=3`: 1 most-images page + 2 fewest-images pages, ~4 min) because the full corpus (~3.7k pages) is too slow for a gate. Run the exhaustive full-corpus version manually with `pnpm --filter @workspace/api-server run verify:cms-io:full`. **The full run is batched** (`checkBatch` per batch of `CMS_IO_BATCH_SIZE` pages, default 200) — it loads + serializes (5 formats) + round-trips one batch at a time so peak memory stays roughly flat as the blog grows instead of materializing the whole corpus at once, and the no-image/link/metadata-loss assertion now spans **every** page processed (not just the sample). Tune memory vs. overhead with `CMS_IO_BATCH_SIZE=N`.
- CMS cross-tab e2e regression (Playwright): `artifacts/cms/e2e/media-alt-sync.spec.ts` + `artifacts/cms/playwright.config.ts` — two REAL browser tabs share one context to prove the bulk alt-text pass's cross-tab sync (the real `storage` event can only be exercised with two live pages); only the media API boundary is stubbed, auth hits the real server via the session `e2e/global-setup.ts` seeds. **Registered as the `cms-e2e` validation step** (in `.replit`) so a dropped cross-tab wire (e.g. a missing `onApprovedChange`/`initialApproved`) fails the gate instead of shipping silently. The config is **self-contained**: `webServer` boots the api-server (localPort 8080) + CMS vite (localPort 23740) on the ports the proxy routes to and polls them *through* `localhost:80`, so the suite needs no pre-started workflows; `reuseExistingServer` makes it reuse running dev workflows when present. Run by hand with `pnpm --filter @workspace/cms run test:e2e`. The bundled Playwright Chromium lacks system libs here — `REPLIT_PLAYWRIGHT_CHROMIUM_EXECUTABLE` (set by the env) points the config at the Nix Chromium.
- TypeScript typecheck gate: **Registered as the `typecheck` validation step** (in `.replit`) so type errors can't ship silently. The api-server (and the scripts jobs) are bundled with esbuild, which does **no** typechecking — a bad type only surfaces at runtime unless something runs `tsc`. The validation runs `pnpm run typecheck:libs && pnpm --filter @workspace/api-server --filter @workspace/cms --filter @workspace/blog run typecheck`: it builds the composite libs first (so their emitted `.d.ts` are fresh for the leaf checks), then typechecks the three artifacts the blog product ships. Scope is intentionally the blog product (api-server + cms + blog); the mobile app (`thanksgiving-mobile`) and `mockup-sandbox` are deliberately excluded so unrelated churn in those surfaces can't red-flag the blog's gate (run the whole repo with `pnpm run typecheck` by hand when you need everything). A failing validation run = a real TypeScript error in the shipped blog/CMS/API code.
- Payload CMS export: `scripts/src/export-payload.ts` + pure mapping in `scripts/src/payload/mapping.ts` (run `pnpm --filter @workspace/scripts run export:payload`; writes `scripts/out/payload-export.json`, gitignored). See `scripts/src/payload/README.md` for the export shape and an example Local-API loader. `buildExport({ pageIds })` restricts the export to a sample of pages, pruning media/authors/categories (incl. ancestors)/tags to only referenced docs; default (no opts) is unchanged full export. `importExport(collections, executor?)` accepts an injected DB executor (e.g. a transaction) for non-destructive imports.
- Real-data round-trip verification (opt-in): `scripts/src/payload/__tests__/roundtrip-real-data.test.ts` — runs the REAL exporter against the live DB on a small interesting page sample (most + fewest images), loads it into an ephemeral Payload, and re-imports inside a rolled-back transaction, asserting no inline-image/link/metadata loss at each hop. Gated on `VERIFY_REAL_DATA=1` so the normal suite skips it. Run with `pnpm --filter @workspace/scripts run verify:roundtrip`.

## Read API (migration foundation)

All under `/api`. Slugs are the public identifier — internal UUIDs never appear in routing.
- `GET /posts` — paginated list; filters `?category=`, `?author=`, `?tag=` (slugs), `?page=`, `?limit=`.
- `GET /posts/{slug}` — full post: original/cleaned HTML, richText JSON, Payload-style componentTree, breadcrumbs, faq, images, seo, jsonld, categories, tags, author.
- `GET /categories`, `GET /categories/{slug}`, `GET /authors`, `GET /authors/{slug}`, `GET /search?q=`.

### CMS search & saved views (staff-gated)

- `GET /cms/search` — multi-field search (plain case-insensitive **ILIKE only**, no `pg_trgm`/extension) over ALL content fields (title/slug/url, body blocks/CTA, seo title+desc, faq, breadcrumbs, jsonld, internal/external links, author, category, tags). Includes ALL statuses (draft/published/archived). Filters `?q=`, `?status=`, `?pageType=`, `?language=`, `?category=`, `?author=`, `?tag=` (repeatable), `?sort=relevance|title|published|updated|created`, `?page=`, `?limit=`. Requires auth + `content.view`. Logic in `searchCmsPosts`/`buildSearchPredicate` (`artifacts/api-server/src/lib/posts.ts`), route in `routes/search.ts`.
- `GET/POST /cms/saved-views`, `PATCH/DELETE /cms/saved-views/{id}` — per-user persisted views (name, description, opaque `query` jsonb), scoped to `req.user.id`. Table `saved_views` (`lib/db/src/schema/saved-views.ts`); routes in `routes/cms-views.ts`. CMS UI: `artifacts/cms/src/pages/search.tsx` (nav-gated on `content.view`).
- **PROD requirement**: `/cms/search` needs only the `saved_views` table (`saved-views.ts`) — it reaches **production** via the normal **re-publish** dev→prod schema diff (the agent must NOT run DDL against prod; prod `executeSql` is read-only). Search itself is **pure ILIKE with no `pg_trgm` extension and no trigram indexes** — they were removed because a GIN index declared on a text column without the trigram operator class made the publish dev→prod diff fail ("GIN index created on a text column without the required operator class"). Because ILIKE needs no special DB setup, there is **no extension, no GIN index, and no self-heal step** for search anymore, and search can't silently break after a dev rollback. Do NOT reintroduce `pg_trgm`/trigram GIN indexes/`ensure:search-indexes`/a search-readiness probe — it re-breaks publishing.
- **Scheduler health probe**: the in-process 60s auto-publish scheduler can silently stall in prod (interval lost / repeated tick errors), leaving `scheduled` posts unpublished with no alert. `artifacts/api-server/src/lib/scheduler-health.ts` mirrors the publishing/analytics readiness pattern: the loop in `index.ts` records each tick (success via `recordSchedulerSuccess`, failure via `recordSchedulerError`), `checkSchedulerHealth` combines that in-memory tick liveness with a DB count of overdue `scheduled` posts (`scheduledFor` older than `OVERDUE_THRESHOLD_MS`=5m), and the result is both logged at startup (`probeSchedulerHealth`, a WARN when unhealthy) and pollable at `GET /api/healthz/scheduler` (200 healthy, 503 when the tick is stale past `TICK_STALENESS_MS`=3×interval / last tick errored / posts overdue, 500 only if the DB probe itself fails). `SCHEDULER_INTERVAL_MS` is the single source of truth shared by the loop and the staleness threshold. The standalone `publish:scheduled:prod` scheduled deployment remains the backstop. Pure `evaluateSchedulerHealth` is unit-tested (`artifacts/api-server/src/lib/__tests__/scheduler-health.test.ts`). Not in the OpenAPI/orval contract (like the sitemap/feed routes).
- **Publishing readiness probe**: mirrors the scheduler/analytics readiness probes for the CMS publishing/scheduling lifecycle. The api-server logs a clear WARN at startup when the `page_status` enum is missing the `review`/`scheduled` values or `pages.scheduled_for` is absent (so a publish that failed to apply these dev→prod shapes is visible in the deployment logs instead of waiting for a write-time 500 when an editor moves a post to review/scheduled). Operators can also poll `GET /api/healthz/publishing` (200 when ready, 503 with `missingStatusValues` + `scheduledForColumnPresent` otherwise). The required shapes + side-effect-free check live in `@workspace/db` (`lib/db/src/publishing-shapes.ts`: `REQUIRED_PAGE_STATUS_VALUES`, `SCHEDULED_FOR_COLUMN`, `checkPublishingReadiness`) and are kept in lockstep with the `ensure:publishing` self-healing script (`scripts/src/ensure-publishing-shapes.ts`) and the schema (`lib/db/src/schema/{enums,pages}.ts`). Probe wrapper: `artifacts/api-server/src/lib/publishing-readiness.ts`.

## Architecture decisions

- **Crawl mode = HTTP (no browser).** Headout's blog is server-rendered WordPress, so plain HTTP returns the full article HTML. An HTTP-vs-Playwright comparison over representative pages (`scripts/src/crawl-compare.ts`) found identical *editorial* content (text, headings, FAQs, links, images, component tree) at ~100–600× the speed, while a real browser was slow and fragile here (`networkidle` never settles; some pages stall). The only browser-only images are JS-injected commerce/recommendation widgets and social-share icons — not article content — so the full crawl runs with `config.useBrowser=false`. (Bundled Playwright Chromium can't download in this env; a Nix `chromium` via `executablePath` is required for any browser work.)
- **DB target defaults to the Replit-managed Postgres (`DATABASE_URL`).** `lib/db/src/index.ts` and `lib/db/drizzle.config.ts` use `DATABASE_URL` unless `USE_SUPABASE` is truthy *and* a non-empty `SUPABASE_DATABASE_URL` is set — only then do they use Supabase. This explicit opt-in (not the old implicit `SUPABASE_DATABASE_URL ?? DATABASE_URL`) prevents a **stale** `SUPABASE_DATABASE_URL` (which can linger in a long-running workflow's env even after the secret is deleted) from silently hijacking the connection to a paused database. To move to Supabase later: set `SUPABASE_DATABASE_URL` and `USE_SUPABASE=true`.
- **When using Supabase, use its Session Pooler URL** (port 5432, `*.pooler.supabase.com`), not the direct `db.*.supabase.co` host (IPv6-only, unreachable here). SSL detection keys off a `supabase.(co|com)` host in the chosen connection string: `rejectUnauthorized:false` (pooler cert doesn't chain to a public CA); `drizzle.config.ts` appends `sslmode=no-verify` (NOT `require`).
- **Lossless page storage**: `pages` keeps `originalHtml`, `cleanedHtml`, `richText` (JSON) and `componentTree` (Payload-compatible nested blocks) so future parser changes never require recrawling.
- **No nested `/{slug}/posts` endpoints**: orval names the zod value and TS type identically for path+query operations (TS2308). "A category's posts" is served by `GET /posts?category={slug}` instead.
- **Schema push**: use `pnpm --filter @workspace/db run push-force` — interactive `push` hangs without a TTY. **On the Replit-managed Postgres (Helium / `DATABASE_URL`), `drizzle-kit push` also silently dies during the "Pulling schema" introspection step** (no error, nothing applied). Workaround: temporarily add an `out` dir to `drizzle.config.ts`, run `drizzle-kit generate --config ./drizzle.config.ts` (no extra flags), apply the emitted SQL via `executeSql`, then revert the `out` line and delete the generated dir.

## Product

- The root path `/` intentionally serves no app. (The former Thanksgiving Family Destination Guide artifact was removed; its design system lives on in the blog.)
- **Headout Blog** (`artifacts/blog`, served at `/blog/`): a public blog that renders the migrated Headout content from the API (`/api`) via generated React Query hooks (`@workspace/api-client-react`), using its own warm editorial design system (`index.css`, `components/ui`). Pages: article (`/blog/<slug>/`), paginated index (`/blog/`), category (`/blog/category/<slug>`), author (`/blog/author/<slug>`), and search (`/blog/search?q=`). Public slugs only — internal UUIDs never appear in routes. Article body renders from `componentTree` → `richText` → `contentHtml` fallback.
- **Blog Studio CMS** (`artifacts/cms`, slug `cms`, served at `/cms/`): an internal, authenticated content-management app on the blog's "Eevee" design system (shares `index.css` + `components/ui`). Auth is **Replit Auth** (OIDC) via the shared `/api` server — `useAuth` (`@workspace/replit-auth-web`) for the session, layered with `GET /cms/me` for role + effective permissions (`src/lib/cms-auth-context.tsx`). RBAC is centralized in `@workspace/cms-auth` (7 roles: admin/editor/writer/seo/reviewer/translator/viewer; `ROLE_PERMISSIONS`, `hasPermission`, `ROLE_META`) and used by BOTH server middleware (`artifacts/api-server/src/middlewares/rbac.ts`) and client nav/route gating (`AppShell`, `RequirePermission`). The **first-ever** signed-in user is bootstrapped to `admin` (`upsertUser`, role only set on INSERT); everyone else defaults to `viewer`. Privileged writes (role changes) append to `audit_logs` via `recordAudit` (`artifacts/api-server/src/lib/audit.ts`). Shell: top nav + side nav + user menu w/ sign-out, home placeholder, and an admin-only Users screen (`PATCH /cms/users/{userId}/role`). **Note:** the `cms` dev workflow can report `DIDNT_OPEN_A_PORT` (newly-created-artifact probe gotgotcha) even though Vite serves `/cms/` fine — reload the workspace or publish.

## Gotchas

- **`componentTree` has two shapes.** The crawler (`crawler/assemble.ts`) stores `pages.componentTree` as a top-level JSON **array** of blocks; the importer (`import/parse.ts`) stores a single root **object**. The `componentTree` field in `openapi.yaml` must stay a `oneOf` (object/array/null) or `GET /posts/{slug}` 500s on every crawler-ingested page. Regenerate zod after any contract change. **They ALSO differ in the block discriminator key:** crawler nodes use `type` (its own `crawler/normalize.ts` builder), the importer/shared `@workspace/content` use `blockType`. The shared `componentTree` renderer (`artifacts/blog/src/components/blocks/renderer.tsx`) handles BOTH by checking for the presence of the discriminator key on the node or its first child. Never flatten the spec to a single shape without migrating all existing data.
